# Strange Path Agent Notes

These notes are the working memory for agents contributing to `strange-path`.
Read them before making engine, gameplay, map, or worldbuilding changes.

## Load Order

1. `project-vision.md` - what kind of game this is.
2. `engine-capabilities.md` - what the engine must support.
3. `rendering-ui.md` - canvas rendering and accessible UI boundaries.
4. `debug-performance.md` - diagnostics, profiling, and performance rules.
5. `demo-scope.md` - what the first playable demo should prove.
6. `world-rules.md` - world, storytelling, and map rules.
7. `implementation-rules.md` - repo and coding constraints for agents.

## Current Intent

`strange-path` is a pixel-art soulslike about facing enemies and bosses across a
huge, mysterious map. The story is not explained directly. The player should
learn through exploration, landmarks, enemy placement, item context, NPC hints,
and locked or unreachable paths.

Before building the full world, the project needs a small engine demo in
`packages/engine-demo`. The first demo should stay intentionally narrow: one
area, one moving character, basic sound, and ray-based lighting/visibility. It
exists to test engine feel and diagnostics before combat, bosses, or map scope
expand.

The game should render through canvas, not React. Menus and other user
interaction surfaces belong in a separate engine UI module that can use native
HTML elements for keyboard accessibility, focus behavior, and assistive
technology support.

## Open High-Level Decision

The camera and movement perspective is not finalized yet:

- Top-down or three-quarter 2D view.
- Side-view action platformer.
- Isometric or another hybrid.

Do not overbuild perspective-specific systems until this is decided.

## Non-Goals

Procedural generation is outside the scope of the engine. The world can be huge,
but it should be authored, structured, and debugged through deliberate map and
content tools rather than generated by the runtime engine.
